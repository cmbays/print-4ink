Competitors Analyzed

Printavo
Market leader ($109-399/mo, 3,000+ shops). Calendar-centric, 13 customizable statuses, dual dates, preset task lists. Gap: No true Kanban board (gated behind $399 Premium), no capacity planning, no QC gates, no production analytics.
PrintLife
Indie product by former shop owner. 4-lane fixed Kanban, quote=invoice conflation, customer portal strength. Gap: No sub-stages, no urgency indicators, single-user only, no service type visibility.

Board Architecture (User-Designed)

Through the 12-question interview, the user collaboratively designed the entire board architecture. Key decisions:

Ready In Progress Review Blocked Done +------------+--------------+------------+--------------+------------+ Quotes | scratch | drafting | customer | waiting on | accepted | | new leads | building | reviewing | customer | auto-clear | +------------+--------------+------------+--------------+------------+ Jobs | approved | screen prep | QC check | blanks not | shipped | | not started| printing | sign-off | arrived | awaiting | | | embroidery | | art issue | payment | +------------+--------------+------------+--------------+------------+
01
Board is single source of truth — Today is a filter, not a dashboard
02
Universal lanes for all service types — Ready / In Progress / Review / Blocked / Done
03
Service type = primary visual — color + icon, instantly scannable
04
Cards are command centers — tasks, actions, history, links
05
Quick capture over forced structure — scratch notes for lightweight logging
06
Guardrails, not gates — quality checkpoints that help without burdening
07
Conservative warnings only — never false positives on capacity
08
Inferred intelligence — productivity from state transitions, not manual logging

Job Lane State Machine

stateDiagram-v2
    [*] --> ready: Job created (from quote or direct)
    ready --> in_progress: Start work
    in_progress --> review: Tasks complete, QC check
    in_progress --> blocked: External dependency
    review --> done: QC passed
    review --> in_progress: QC failed (return with note)
    blocked --> in_progress: Unblocked
    blocked --> ready: Unblocked (not started)
    done --> [*]: Archived after 7+ days

Implementation Plan — 5-Wave Build Strategy

The build is organized into 5 dependency-ordered waves. Each wave produces a demoable checkpoint, isolates risk, and allows quality gates between phases. Wave 4 parallelizes two independent screens for efficiency.

Source: docs/strategy/jobs-implementation-plan.md

Wave Summary

Wave Name Objective Agent Sessions
1 Foundation Schema + Constants + Mock Data + Tests general-purpose 1 (serial)
2 Board Core Static Board Layout + Card Components + Filters frontend-builder 1 (serial)
3 Board DnD Drag-and-Drop + Dialogs + Scratch Note Capture frontend-builder 1 (serial)
4A Job Detail Command Center — 7 sections, full interactivity frontend-builder 1 (parallel)
4B Jobs List DataTable with sort/filter/search/quick actions frontend-builder 1 (parallel)
5 Integration Cross-vertical wiring, dashboard, polish, audit frontend-builder + design-auditor 1 (serial)

PR Strategy (Sequential Merge)

Each wave merges to main independently, in strict order:

Merge order: Wave 1 → Wave 2 → Wave 3 → Wave 4A + 4B (either order) → Wave 5

Waves 4A and 4B create different files and can merge in either order. Wave 5 must merge last since it touches files from all previous waves. Each wave branch is based on main, not stacked.

Quality Gate Protocol (19-Point Checklist)

Every wave runs through a 19-point quality gate covering 4 categories:

Category Checks Items
Build Checks Automated TypeScript, ESLint, Build, Tests
Visual Checks Manual Card design, board layout, detail sections, table columns, responsive
Cross-Link Checks Manual Route references, breadcrumbs, cross-vertical links
Accessibility Manual ARIA labels, keyboard navigation, focus management
Design System Manual Color tokens only, Lucide icons only, Inter font, Tailwind spacing

Formal design audits run at the end of Wave 2 (board visual review) and Wave 5 (final comprehensive audit).

Risk Register

# Risk L I Mitigation
1 dnd-kit 2-section board complexity M H Spike if stuck >2h. Fallback: single-section first. Composite droppable IDs.
2 Board performance with many cards L M 15-20 max mock cards. useMemo + React.memo. Virtualize in Phase 2.
3 Cross-vertical breakage from schema changes M H Keep old productionStateEnum exported. Full build after Wave 1.
4 Schema migration breaks existing tests M L Rewrite job.test.ts in Wave 1 before UI work. Run all 314+ tests.
5 Task state management complexity M M Simple array on job state. No separate store. Pure functional updates.
6 Wave 4 parallel merge conflicts L L 4A and 4B create different files. No shared file modifications.
7 shadcn/ui progress component missing L L Check in Wave 2. Install if needed, or build simple div-based bar.
8 dnd-kit package not in project L L Check package.json at Wave 3 start. npm install if needed.
